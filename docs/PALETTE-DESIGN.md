# Palette Design for omarchy-emacs-themer

## How the themer maps a palette to Emacs faces

`20-emacs.sh` extracts colors from a theme's `colors.toml` (or falls back to
`alacritty.toml`) and maps them to Emacs face attributes. Understanding this
mapping is important both for evaluating whether a theme will look good and for
troubleshooting when it does not.

### Color slots and their semantic roles

| Source variable | Emacs alias | Used for |
|---|---|---|
| `foreground` | `fg` | Default text, active mode-line text (dark) |
| `background` | `bg` | Default background, inactive mode-line background (light) |
| `cursor` | `cursor-bg` | Cursor fill |
| `selection_foreground` | `sel-fg` | Selected text, active mode-line text (light), LSP popup text |
| `selection_background` | `sel-bg` | Selected region, active mode-line background (light), LSP popup background |
| `color0` (normal black) | `black` | Active mode-line background (dark), vertical borders |
| `color7` (normal white) | `white` | Dim backgrounds |
| `color8` (bright black) | `br-black` | Comments, line numbers, inactive mode-line text |
| `color1–6` | `red`…`cyan` | Syntax highlighting |
| `color9–14` | `br-red`…`br-cyan` | Bright syntax variants |

### Light vs dark theme handling

The presence of a `light.mode` file in the theme directory triggers different
face mappings for UI elements that need to adapt to a light background:

| Element | Dark theme | Light theme |
|---|---|---|
| Active mode-line background | `black` (`color0`) | `sel-bg` |
| Active mode-line text | `fg` | `sel-fg` |
| Inactive mode-line background | `black` (`color0`) | `bg` |
| Inactive mode-line text | `br-black` (`color8`) | `br-black` (`color8`) |
| Mode-line emphasis text | `blue` (`color4`) | `sel-fg` |
| LSP popup background | `black` (`color0`) | `sel-bg` |
| LSP popup text | `fg` | `sel-fg` |

## What makes a palette work well with this themer

For the themer to produce good results, a palette needs **tonal diversity** —
the key color roles must be clearly separated in lightness and hue so that text
is readable against whatever background it appears on.

### Essential properties

**1. Clear fg/bg separation**
`foreground` and `background` must have strong contrast. This is the minimum
requirement for any usable color scheme and almost all themes satisfy it.

**2. `sel-bg` must be tonally distinct from BOTH `bg` AND `fg`**
This is the most commonly violated property. `sel-bg` is used as:
- the active mode-line background on light themes
- the LSP popup body background on light themes
- the region highlight color everywhere

If `sel-bg` is tonally close to `fg`, text sitting on `sel-bg` (which inherits
color from `fg` or from syntax faces derived from the ANSI palette) will have
poor contrast. For dark themes the equivalent requirement applies to `color0`.

**3. `sel-fg` must contrast clearly against `sel-bg`**
These two are designed as a pair and theme authors usually get this right, but
it is worth verifying. The themer uses `sel-fg` as the popup and mode-line text
color on light themes precisely because this relationship is guaranteed by the
theme author.

**4. ANSI colors should span multiple hue families**
`color0`–`color15` are used for syntax highlighting. If all of them are in the
same hue family (all teal, all warm, all blue-gray), some syntax faces will be
indistinguishable from each other and from the backgrounds used for UI panels.

**5. For light themes: `color0` should be distinctly dark**
On dark themes, `color0` (normal black) serves as the dark panel color for the
mode-line. On light themes we switch to `sel-bg`, but the ANSI colors still
appear in syntax highlighting across the editor. If they cluster around the
same mid-tone as `sel-bg`, syntax faces on modeline segments (which doom-modeline
renders using font-lock faces) will have poor contrast.

## The palette bias problem

Some themes — particularly those generated by automated tools like the Aether
theme builder — produce palettes that are heavily weighted toward a single hue
family. Every color in the palette is a variant of the same base hue
(teal, warm earth, cool blue, etc.) with only small differences in lightness
and saturation.

These palettes cause cascading contrast failures in this themer because:

- `sel-bg` and `fg` are similar in tone → poor contrast on mode-line and popups
- ANSI syntax colors are all in the same family → syntax faces in mode-line
  segments (doom-modeline reuses `font-lock-string-face` for paths and branch
  names) blend into the mode-line background
- There is no "neutral" fallback color we can choose algorithmically that is
  both tonally distinct from the biased palette AND in keeping with the theme

### Why there is no generic fix

The themer picks UI background colors from the palette (`sel-bg`, `black`)
because those are the colors the theme author intended for highlighted/panel
contexts. For a well-designed palette this works. For a heavily biased palette,
every candidate background color is too similar to every candidate foreground
color.

Computing contrast ratios in bash and falling back to an alternative is
theoretically possible but has no reliable fallback: if all palette colors are
teal, no teal is sufficiently distinct from any other teal.

## Per-theme overrides (future direction)

The right solution for problematic palettes is a per-theme escape hatch: if a
theme ships (or the user places) an `emacs-overrides.toml` file alongside its
other config files, `20-emacs.sh` would read it and substitute those values
before generating the theme.

A minimal override file might look like:

```toml
# Override specific color roles for Emacs only.
# Any key here replaces the value extracted from colors.toml/alacritty.toml
# before the theme file is generated.
#
# Useful when a theme's palette is tonally biased and the automatically
# chosen UI colors (mode-line, popup backgrounds) have poor contrast.

modeline_bg  = "#2e3440"   # use a neutral dark instead of sel-bg
modeline_fg  = "#eceff4"
popup_bg     = "#2e3440"
popup_fg     = "#eceff4"
```

The override file would live alongside the theme:
```
~/.config/omarchy/themes/my-aether-theme/
├── colors.toml
├── emacs-overrides.toml   ← user-authored
└── ...
```

Or for system themes the user cannot modify directly, in a user config directory:
```
~/.config/omarchy/emacs-overrides/
└── my-aether-theme.toml
```

This has not been implemented yet. If you are working with a palette-biased
theme today, the practical workaround is to add face customizations in your
Doom config after the theme loads:

```elisp
(defun my/fix-aether-faces ()
  (when (memq 'omarchy-doom custom-enabled-themes)
    (set-face-attribute 'mode-line nil
                        :background "#2e3440"
                        :foreground "#eceff4")
    (set-face-attribute 'lsp-ui-doc-background nil
                        :background "#2e3440"
                        :foreground "#eceff4")))

(add-hook 'after-load-theme-hook #'my/fix-aether-faces)
```
